# Make Documentation Decision Workflow

Make a documentation decision before creating or changing docs.

## When To Use

- Deciding whether to create, update, merge, split, defer, or remove docs.
- Choosing between a quick patch and a larger doc plan.
- Resolving unclear ownership, audience, source material, or release timing.
- Communicating a recommendation to PM, engineering, support, or docs leads.

## Prerequisites

- Decision question or proposed documentation change.
- Product context, audience hypothesis, release timing, or existing docs when available.

**Reference**: from this workflow file, open `../references/core/knowledge.md`.

## Workflow Steps

### Step 1: State The Decision

**Goal**: Make the actual choice visible.

- [ ] Write the decision as a question or recommendation.
- [ ] Identify the affected product surface and audience.
- [ ] Identify the user goal, risk, or confusion the decision should address.

### Step 2: Gather Evidence

**Goal**: Avoid deciding from internal preference alone.

- [ ] Check existing docs, release notes, support issues, analytics, search terms, and feedback.
- [ ] Identify product constraints, release timing, review needs, and available writers.
- [ ] Mark assumptions separately from verified facts.

### Step 3: Compare Options

**Goal**: Choose a defensible path.

- [ ] List realistic options, including doing nothing.
- [ ] Compare impact on users, maintenance load, timing, and accuracy risk.
- [ ] Identify dependencies or follow-up work each option creates.

### Step 4: Recommend And Communicate

**Goal**: Turn the decision into an executable plan.

- [ ] Recommend one option and explain why.
- [ ] Name owner, reviewers, and next action.
- [ ] Share open questions and the date or trigger for revisiting the decision.

## Exit Criteria

- [ ] The decision, evidence, recommendation, owner, and open questions are explicit.
- [ ] The recommendation maps to user impact, not internal completeness alone.
- [ ] The next documentation action is clear.
