# Integrate Docs With Agile Workflow

Plan documentation work inside Agile product development without hiding docs-specific needs.

## When To Use

- Documentation is missing from sprint planning, release planning, or definition of done.
- Writers need to track feature docs, reviews, or blockers in the team's Agile tooling.
- Docs work spans multiple sprints, scrum teams, or release trains.
- A team needs lightweight docs checkpoints without heavy process.

## Prerequisites

- Team process, release cadence, Agile board, feature tickets, or planning artifacts.
- Product owner, PM, engineering, QA, and docs contacts when available.

**Reference**: from this workflow file, open `../references/core/knowledge.md`.

## Workflow Steps

### Step 1: Map The Team Cadence

**Goal**: Understand how docs should attach to product work.

- [ ] Identify sprint length, release timing, planning meetings, demos, and retrospectives.
- [ ] Identify where feature decisions, review requests, and defects are tracked.
- [ ] Identify whether the docs work is feature-scoped or cross-feature.

### Step 2: Make Docs Work Visible

**Goal**: Put documentation into the team's normal planning system.

- [ ] Add or link docs tasks to feature work.
- [ ] Add story points, priority, owner, blockers, and review needs when the team uses them.
- [ ] Track docs questions, review requests, and defects in the same system engineers use.

### Step 3: Define Done For Docs

**Goal**: Align release readiness with user-facing documentation.

- [ ] Add docs impact assessment to planning.
- [ ] Add draft, technical review, QA validation, publish, and release-note checks where relevant.
- [ ] Identify docs work that cannot finish in the same sprint as code and plan a realistic handoff.

### Step 4: Use The Right Process For The Work

**Goal**: Avoid forcing every docs activity into a feature sprint.

- [ ] Use scrum closely for feature-specific docs.
- [ ] Use a separate docs project, checklist, or Kanban lane for troubleshooting, maintenance, scenarios, learning objectives, or cross-team docs.
- [ ] Preserve what the team needs from sprint work even when the docs project spans multiple sprints.

### Step 5: Inspect And Adapt

**Goal**: Improve the process from evidence.

- [ ] Bring docs blockers and review misses to retrospectives.
- [ ] Check whether customer feedback or support trends show docs were late, unclear, or incomplete.
- [ ] Adjust intake, review timing, templates, or ownership.

## Exit Criteria

- [ ] Docs work is visible in the product team's planning system.
- [ ] Definition of done includes user-facing documentation where relevant.
- [ ] Docs-specific work has a cadence that matches its real scope.
