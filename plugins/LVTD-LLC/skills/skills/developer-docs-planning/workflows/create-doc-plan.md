# Create Doc Plan Workflow

Create a practical documentation plan for a developer-facing feature, product, or doc set.

## When To Use

- Planning docs before a feature release.
- Turning audience research into a doc outline.
- Reviewing whether an existing doc set covers user needs.

## Prerequisites

- Audience or user goal hypothesis.
- Product surface, feature brief, API schema, release plan, or existing docs.

**Reference**: from this workflow file, open `../references/core/knowledge.md`.

## Workflow Steps

### Step 1: Establish Scope

**Goal**: Define what the plan must cover.

- [ ] Name the product, feature, or doc set.
- [ ] Identify primary audience and first successful outcome.
- [ ] Identify release timing, owner, and review constraints.
- [ ] List existing docs that should be reused, updated, or retired.

### Step 2: Choose Content Types

**Goal**: Match docs to reader tasks.

- [ ] List user questions and tasks.
- [ ] Pick the smallest set of content types that answers them.
- [ ] Split learning, task, reference, troubleshooting, and announcement needs.
- [ ] Avoid creating a doc for internal completeness alone.

### Step 3: Build The Outline

**Goal**: Make missing information visible before drafting.

- [ ] Create one row per doc or section.
- [ ] Include purpose, audience, source of truth, owner, reviewers, and dependencies.
- [ ] Mark prerequisites, versions, access requirements, and known caveats.
- [ ] Identify open questions that block accurate drafting.

### Step 4: Plan Review And Maintenance

**Goal**: Keep the plan executable after launch.

- [ ] Assign technical and editorial reviewers.
- [ ] Identify generated or authoritative sources.
- [ ] Define release/publishing timing.
- [ ] Define likely quality or feedback signals.

### Step 5: Report The Plan

**Goal**: Give the team a usable artifact.

- [ ] Summarize recommended docs and rationale.
- [ ] Include a table of planned docs.
- [ ] Call out risks, blockers, and decisions needed.
- [ ] Recommend immediate next actions.

## Exit Criteria

- [ ] Each planned doc maps to a real user task or release obligation.
- [ ] Each doc has a source of truth and owner.
- [ ] The plan exposes blockers rather than hiding them in the draft.
