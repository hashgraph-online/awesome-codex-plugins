# Plan Scenario Docs Workflow

Plan scenario-driven developer documentation around a real user problem.

## When To Use

- Users need an end-to-end example that connects concepts and tasks.
- Existing docs are accurate but do not show a realistic workflow.
- Product, support, field, training, or marketing teams are aligning around a common use case.
- A feature needs context beyond a basic tutorial but should not become comprehensive docs.

## Prerequisites

- Audience hypothesis.
- Candidate scenario, customer problem, product workflow, or user story.
- Access to product, sample data, SMEs, QA, or customer-facing teams when available.

**Reference**: from this workflow file, open `../references/core/knowledge.md`.

## Workflow Steps

### Step 1: Define Audience And Goal

**Goal**: Keep the scenario from becoming generic.

- [ ] Name the specific audience or persona.
- [ ] State the real-world problem and desired outcome.
- [ ] Identify the starting point, prerequisites, permissions, and sample data.

### Step 2: Gather Candidate Scenarios

**Goal**: Source scenarios from real use.

- [ ] Review PM user stories and feature intent.
- [ ] Ask support, field, education, sales, and community teams what users are trying to do.
- [ ] Reframe feature-driven workflows as customer problems.
- [ ] Eliminate scenarios that the product does not support cleanly.

### Step 3: Design The Path

**Goal**: Give readers a seamless route to the solution.

- [ ] Outline the end-to-end workflow in user order.
- [ ] Include only concepts, steps, and options needed for this scenario.
- [ ] Link to reference or advanced material instead of covering every variation.
- [ ] Identify screenshots, sample files, realistic data, or media that will add maintenance cost.

### Step 4: Validate The Scenario

**Goal**: Confirm that the scenario is useful and supported.

- [ ] Review with PM, QA, support, and customer-facing teams.
- [ ] Test the workflow with available product state and realistic data.
- [ ] Ask whether the product or UX should change if the scenario is awkward.
- [ ] Validate with customers when possible.

### Step 5: Hand Off To Drafting

**Goal**: Make the scenario ready to write.

- [ ] Provide title, audience, goal, prerequisites, outline, and validation notes.
- [ ] Mark scope boundaries and related docs.
- [ ] Name technical reviewers and unresolved facts.

## Exit Criteria

- [ ] The scenario maps to a real audience problem.
- [ ] The workflow is supported, testable, and scoped.
- [ ] Adjacent concepts, variations, and references are linked rather than overexplained.
