# Beta Reader Feedback Rules

Use these rules when setting up beta reading or synthesizing beta feedback.

## Core Rules

### 1. Start Before The Manuscript Feels Finished

Begin beta reading when the draft is coherent enough to follow, not when it is polished.

- Structural feedback is easier before the author is attached to wording.
- Rougher drafts make readers more willing to name big problems.
- Polishing before beta reading wastes time if the structure changes.

### 2. Recruit More Readers Than You Need

Many volunteers will not engage.

- Aim for a small set of deeply engaged readers per round.
- Invite more people than the desired final count.
- Use fresh readers for new iterations when possible.

### 3. Use A Live Commenting Surface

Avoid feedback workflows that hide engagement.

- Do not rely on emailed PDFs as the main process.
- Use comments or suggestions directly in a shared manuscript.
- Preserve comment location so signals map to exact sections.

### 4. Tell Readers What Feedback Matters

Give instructions that match the manuscript stage.

- Early: ask for confusion, boredom, missing value, skepticism, and major structure issues.
- Later: ask for accuracy, examples, clarity, polish, and endorsements.
- Tell readers not to spend the round only fixing typos unless that is the current goal.

### 5. Classify Before Revising

Group comments by signal before editing.

- Value: preserve, emphasize, repeat.
- Confusion: clarify, reorder, add prerequisite, or simplify.
- Skepticism: add proof, caveat, example, or soften claim.
- Boredom: cut, move value earlier, or increase value density.
- Inaccuracy: verify and correct.
- Drama: delete or de-escalate.

### 6. Treat Abandonment As Data

Do not pressure readers through the whole book.

- Where readers stop is a signal.
- Surrounding sections deserve extra review.
- Later rounds should show readers getting farther through the manuscript.

### 7. Do Not Argue With Readers

If a reader gets stuck, future readers may too.

- Step away before responding defensively.
- Use feedback as a lens into the manuscript problem.
- Fix the manuscript, not the individual reader.

## Guidelines

- Save influential readers for later rounds when the manuscript is stronger.
- Ask for testimonials when influential readers voluntarily signal strong value.
- Follow up after reading when the book is meant to change real-world behavior.
- Start pre-sales only when the structure and knowledge are mostly working.

## Exceptions

- **Expert review**: Experts may be reused for targeted accuracy checks.
- **Sensitivity or legal review**: These can happen late and may use specialized readers outside the target reader profile.
- **Tiny audience**: If fresh readers are scarce, be explicit about how repeat-reader feedback is less reliable.

## Quick Reference

| Rule | Summary |
|------|---------|
| Start rough | Beta reading is for structural learning. |
| Over-invite | Only some readers will engage deeply. |
| Live comments | Location and drop-off are data. |
| Give instructions | Match feedback type to manuscript stage. |
| Classify first | Convert comments into signal categories. |
| Abandonment data | Stopping points reveal friction. |
| No arguing | Fix the manuscript instead. |
