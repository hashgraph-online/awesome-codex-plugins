# Experiment Type Selection Examples

## Superiority Scenario

**Situation**: A new personalized module is expected to increase content plays.

**Recommended type**: Superiority.

**Why**: The feature is justified by improved engagement. If engagement does
not improve and guardrails do not compensate, the launch rationale is weak.

## Non-Inferiority Scenario

**Situation**: A team replaces a legacy checkout component with a more
maintainable implementation. The new version should not reduce checkout
completion beyond an agreed margin.

**Recommended type**: Non-inferiority.

**Why**: The value is maintainability and future velocity. The experiment only
needs to show that business performance is not materially harmed.

## Equivalence Scenario

**Situation**: Two recommendation layouts have similar implementation cost and
the design team is comfortable with either if user behavior is practically the
same.

**Recommended type**: Equivalence.

**Why**: The decision requires proving sameness within an acceptable range, not
finding a winner.

## A/B/n Scenario

**Situation**: Three CTA placements differ only in location. Copy, color, and
surrounding layout are unchanged.

**Recommended type**: A/B/n if traffic supports it.

**Why**: Each variant isolates the same variable, so comparison can produce
interpretable learning.

## Holdback Scenario

**Situation**: A recommendation system is launched broadly, but the team needs
to know whether it changes retention or consumption over months.

**Recommended type**: Long-term holdback.

**Why**: The key outcome may accumulate over time after the launch window.
