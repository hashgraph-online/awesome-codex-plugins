# Map Doc Audience Workflow

Create an audience brief for developer documentation.

## When To Use

- Planning new API, SDK, CLI, onboarding, migration, or troubleshooting docs.
- Reworking docs that feel too internal, too shallow, or too advanced.
- Preparing a documentation plan before writing.

## Prerequisites

- Product, feature, or doc surface to analyze.
- Any available support tickets, analytics, user feedback, sales notes, forum posts, or existing docs.

**Reference**: from this workflow file, open `../references/core/knowledge.md`.

## Workflow Steps

### Step 1: Define The Documentation Outcome

**Goal**: Make the reader's success measurable.

- [ ] Name the product surface or doc set.
- [ ] Write the reader's first useful outcome.
- [ ] Write the business or team outcome the docs support.
- [ ] Mark whether each goal is verified or assumed.

### Step 2: Identify Primary And Secondary Users

**Goal**: Avoid treating all developers as one audience.

- [ ] List likely roles and tasks.
- [ ] Choose the primary audience for this doc.
- [ ] Capture relevant tools, languages, environments, permissions, and prior knowledge.
- [ ] Note secondary audiences that should be routed elsewhere.

### Step 3: Gather User Questions

**Goal**: Turn audience knowledge into doc requirements.

- [ ] List setup and prerequisite questions.
- [ ] List concept and decision questions.
- [ ] List task, API, config, and troubleshooting questions.
- [ ] Flag unanswered questions that need research.

### Step 4: Create The Right Artifact

**Goal**: Preserve the research in a reusable form.

- [ ] Use a persona for a recurring user type.
- [ ] Use user stories for task-focused requirements.
- [ ] Use a journey map for multi-step onboarding or migration.
- [ ] Use a friction log when setup or first-use blockers are unclear.

### Step 5: Validate And Apply

**Goal**: Make the research change the docs.

- [ ] Identify evidence still needed.
- [ ] Recommend interviews, support-ticket review, analytics, or product walk-throughs.
- [ ] Explain how the findings change content type, order, examples, prerequisites, or measurement.

## Exit Criteria

- [ ] The primary audience is specific enough to guide writing decisions.
- [ ] Goals, questions, assumptions, and validation steps are documented.
- [ ] The next documentation planning step is clear.
