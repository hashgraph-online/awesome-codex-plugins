# Define Learning Objectives Workflow

Define audience-specific learning objectives for developer documentation.

## When To Use

- Planning topic scope before drafting.
- Checking whether a page has a clear starting point and destination.
- Splitting overloaded content into modular topics.
- Aligning docs with SaaS roles, permissions, or managed-service boundaries.

## Prerequisites

- Audience or audience hypothesis.
- User goal, feature brief, existing topic, or support signal.

**Reference**: from this workflow file, open `../references/core/knowledge.md`.

## Workflow Steps

### Step 1: Name The Real-World Goal

**Goal**: Separate the user's practical outcome from the learning objective.

- [ ] State the real-world user goal.
- [ ] Identify the primary audience and any role or permission boundary.
- [ ] Mark whether the goal is verified by user evidence or assumed.

### Step 2: Establish The Starting Point

**Goal**: Make the topic entry condition explicit.

- [ ] List prior knowledge, tools, access, environment, and version assumptions.
- [ ] Identify what the reader probably knows already.
- [ ] Identify what the reader must know before the first action or decision.

### Step 3: Choose Objective Type

**Goal**: Match the objective to the reader outcome.

- [ ] Use awareness when the reader needs to identify or describe.
- [ ] Use comprehension when the reader needs to compare, decide, or explain tradeoffs.
- [ ] Use applicable skill when the reader needs to complete and repeat a task.

### Step 4: Write Observable Exit Criteria

**Goal**: Make success testable.

- [ ] Use an action verb such as identify, choose, configure, verify, diagnose, migrate, or explain.
- [ ] State what the reader can do by the end.
- [ ] Add a verification signal or decision outcome.
- [ ] Avoid broad verbs such as "understand" unless paired with observable behavior.

### Step 5: Set Scope Boundaries

**Goal**: Keep the topic focused.

- [ ] Include content required to move from starting point to destination.
- [ ] Link out to background, variations, or advanced options that are not on the route.
- [ ] Remove product internals that do not affect reader action or decision.

## Exit Criteria

- [ ] The objective supports a real-world user goal.
- [ ] Starting point, destination, and exit criteria are explicit.
- [ ] The objective is narrow enough to guide topic scope.
