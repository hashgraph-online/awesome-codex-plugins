# Design Technical Visual Workflow

Design or review a screenshot, diagram, or video for developer documentation.

## When To Use

- Adding a visual to a doc.
- Reviewing an existing screenshot, diagram, or video.
- Choosing the right visual format for a technical explanation.

## Prerequisites

- Reader goal, surrounding text, and concept or task to support.
- Product or system details that the visual must represent.

**Reference**: from this workflow file, open `../references/core/knowledge.md`.

## Workflow Steps

### Step 1: Define Purpose

**Goal**: Avoid decorative or redundant visuals.

- [ ] State what the reader should understand or do after seeing the visual.
- [ ] Identify the nearby text the visual supports.
- [ ] Decide whether text alone would be clearer or more maintainable.

### Step 2: Choose Visual Type

**Goal**: Match format to reader need.

- [ ] Choose screenshot for UI location or state.
- [ ] Choose boxes-and-arrows for entities and relationships.
- [ ] Choose flowchart for decisions or process paths.
- [ ] Choose swimlane for responsibility across actors.
- [ ] Choose video only when motion or timing is essential.

### Step 3: Design For Comprehension

**Goal**: Make the visual easy to decode.

- [ ] Use one main idea or level of detail.
- [ ] Add a clear starting point.
- [ ] Use consistent labels, shapes, lines, and colors.
- [ ] Avoid crossed connectors and crowded labels.
- [ ] Split the visual if it becomes dense.

### Step 4: Check Accessibility And Performance

**Goal**: Make the visual usable for more readers.

- [ ] Provide alt text or caption.
- [ ] Do not rely on color alone.
- [ ] Check contrast and small-text legibility.
- [ ] Keep critical commands and values in text.
- [ ] Use efficient formats and reasonable image sizes.

### Step 5: Plan Maintenance

**Goal**: Keep the visual from becoming stale.

- [ ] Store editable source files.
- [ ] Name owner and update trigger.
- [ ] Review visual with surrounding text.
- [ ] Note responsive or dark-mode checks if applicable.

## Exit Criteria

- [ ] The visual has a clear job and supports nearby text.
- [ ] Accessibility, performance, and source-file maintenance are addressed.
- [ ] The visual does not replace copyable critical information.
