# Evaluate Doc Platform Workflow

Evaluate documentation tools or delivery platforms from reader, author, maintenance, and migration requirements.

## When To Use

- Choosing a documentation platform or authoring tool.
- Deciding whether a tool change will solve findability, reuse, versioning, feedback, publishing, or maintenance problems.
- Comparing wiki, docs-as-code, static site, CMS, help center, DITA, API reference, or custom platform options.
- Preparing a recommendation for a team or leadership decision.

## Prerequisites

- Current pain points, target reader experience, author workflow, content types, and candidate options.
- Existing content inventory or migration constraints when available.

**Reference**: `../references/core/knowledge.md`

## Workflow Steps

### Step 1: Define Outcomes

**Goal**: Anchor the decision in reader and team needs.

- [ ] State reader tasks, entry points, search needs, accessibility needs, and version or localization needs.
- [ ] State authoring, review, preview, publishing, and maintenance workflow needs.
- [ ] Identify business constraints: budget, staffing, compliance, support capacity, timeline, and release cadence.

### Step 2: Diagnose Current Pain

**Goal**: Avoid migrating for the wrong reason.

- [ ] Separate tooling limits from IA, ownership, stale content, review, metadata, or staffing problems.
- [ ] List process fixes that are required regardless of platform.
- [ ] Decide whether a tool change is necessary, helpful, premature, or unrelated.

### Step 3: Build Weighted Requirements

**Goal**: Make the comparison defensible.

- [ ] Define reader usability, content model, authoring workflow, integration, maintenance, migration, portability, support, and cost requirements.
- [ ] Mark each requirement as must-have, should-have, nice-to-have, or not relevant.
- [ ] Identify risks that would disqualify an option.

### Step 4: Compare Options

**Goal**: Evaluate tradeoffs rather than feature lists.

- [ ] Score each option against weighted requirements.
- [ ] Identify implementation, training, customization, migration, and support burden.
- [ ] Identify lock-in and export risk.
- [ ] Identify what content cleanup, IA work, or process changes each option would force.

### Step 5: Recommend A Path

**Goal**: Produce a decision the team can act on.

- [ ] Recommend keep, process-first, pilot, migrate, or reject.
- [ ] State assumptions, open questions, and risks.
- [ ] Define pilot content, success measures, owner, timeline, and rollback or exit plan when needed.
- [ ] Document tradeoffs the team accepts.

## Quick Checklist

```text
[ ] Reader and author outcomes are defined
[ ] Process problems are separated from tool problems
[ ] Requirements are weighted
[ ] Options are compared against requirements
[ ] Migration, support, portability, and cost are explicit
[ ] Recommendation includes next action and owner
```

## Exit Criteria

- [ ] Recommendation follows from prioritized requirements.
- [ ] Platform risks and process dependencies are visible.
- [ ] The team has a next step: keep, fix process, pilot, migrate, or reject.
