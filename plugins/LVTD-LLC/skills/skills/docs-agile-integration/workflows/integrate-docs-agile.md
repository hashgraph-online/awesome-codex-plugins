# Integrate Docs With Agile Workflow

Plan documentation work inside Agile product development without hiding docs-specific needs.

## When To Use

- Documentation is missing from sprint planning, release planning, or definition of done.
- Writers need to track feature docs, reviews, or blockers in the team's Agile tooling.
- Docs work spans multiple sprints, scrum teams, or release trains.
- A team needs lightweight documentation checkpoints without heavy process.

## Prerequisites

- Team process, release cadence, Agile board, feature tickets, planning artifacts, and review expectations.
- Product owner, PM, engineering, QA, support, field, or docs contacts when available.

**Reference**: `../references/core/knowledge.md`

## Workflow Steps

### Step 1: Map The Team Cadence

**Goal**: Understand where documentation should attach to product work.

- [ ] Identify sprint length, release timing, planning meetings, demos, retrospectives, and release gates.
- [ ] Identify where feature decisions, review requests, defects, and blockers are tracked.
- [ ] Identify whether docs work is feature-scoped, release-scoped, or cross-feature.

### Step 2: Add Docs Impact Assessment

**Goal**: Catch docs needs before release pressure.

- [ ] Add a docs impact field, checklist item, label, or planning question for user-facing work.
- [ ] Include public docs, release notes, code samples, screenshots, support docs, runbooks, and training where relevant.
- [ ] Define when a no-docs decision is acceptable and who can make it.

### Step 3: Make Docs Work Visible

**Goal**: Put documentation into the team's normal planning system.

- [ ] Add or link docs tasks to feature work.
- [ ] Include owner, priority, source of truth, reviewers, due timing, and blockers.
- [ ] Track docs questions, review requests, and defects where engineers and PMs can see them.

### Step 4: Match Process To Scope

**Goal**: Avoid forcing every docs activity into a feature sprint.

- [ ] Use sprint-linked tasks for feature-specific docs and release notes.
- [ ] Use a docs epic, project, or Kanban lane for scenarios, maintenance, IA, troubleshooting, or cross-team docs.
- [ ] Keep feature links and release gates visible even when docs work spans sprints.

### Step 5: Inspect And Adapt

**Goal**: Improve from evidence.

- [ ] Bring docs blockers, review misses, late release notes, support tickets, and customer confusion to retrospectives.
- [ ] Adjust intake, review timing, definition of done, ownership, or templates.
- [ ] Record decisions for remote or async teams.

## Quick Checklist

```text
[ ] Team cadence and board model mapped
[ ] Docs impact assessment added
[ ] Docs tasks have owner, source, reviewers, due timing, and blockers
[ ] Feature docs and cross-sprint docs have appropriate tracking
[ ] Retrospective signals are defined
```

## Exit Criteria

- [ ] Docs work is visible in the product team's planning system.
- [ ] User-facing changes receive docs impact assessment.
- [ ] Docs-specific work has a cadence that matches its real scope.
