# Audit Doc IA Workflow

Audit or redesign documentation information architecture.

## When To Use

- Reorganizing a docs site or major section.
- Planning a documentation migration.
- Improving findability, navigation, landing pages, or content inventory.

## Prerequisites

- Current docs or sitemap.
- User tasks, audiences, analytics, search data, support signals, or feedback when available.

**Reference**: from this workflow file, open `../references/core/knowledge.md`.

## Workflow Steps

### Step 1: Define User Model

**Goal**: Ground IA in reader behavior.

- [ ] Identify primary audiences and top tasks.
- [ ] Identify common entry points and next steps.
- [ ] Note search terms, support issues, or feedback if available.

### Step 2: Inventory Content

**Goal**: Understand what exists.

- [ ] List pages, purpose, owner, traffic or usage signals, and source of truth.
- [ ] Mark stale, duplicate, missing, misplaced, or overloaded content.
- [ ] Label each page: keep, remove, review, merge, split, move, or create.

### Step 3: Choose Structure

**Goal**: Create a navigable model.

- [ ] Use sequences for onboarding, setup, or migration.
- [ ] Use hierarchies for product or task groups.
- [ ] Use web links for related non-linear tasks.
- [ ] Define landing pages and navigation cues.

### Step 4: Plan Migration

**Goal**: Move content without breaking readers.

- [ ] Map old URLs to new URLs.
- [ ] Define redirects, metadata, labels, and search updates.
- [ ] Update internal links and related links.
- [ ] Identify validation checks after migration.

### Step 5: Maintain IA

**Goal**: Prevent drift after redesign.

- [ ] Document IA decisions.
- [ ] Assign owners or review cadence.
- [ ] Define rules for adding, splitting, merging, and deleting content.

## Exit Criteria

- [ ] Recommended IA maps to user tasks.
- [ ] Content actions and migration plan are explicit.
- [ ] Maintenance decisions are documented.
