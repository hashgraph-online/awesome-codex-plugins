## 审查维度

逐项检查以下三个方面：

### 1. 完整性 — 所有需求都实现了吗？

- 将规格中的每一条需求提取为检查项
- 逐条在变更文件中寻找对应实现
- 标记每条需求的状态：✅ 已实现 / ❌ 未实现 / ⚠️ 部分实现

### 2. 精确性 — 没有额外添加规格之外的东西吗？

- 检查是否存在规格中未要求的功能
- 检查是否引入了超出范围的依赖或模块
- 额外添加不一定是错误，但必须标注出来

### 3. 一致性 — 实现与设计匹配吗？

- API 接口是否与设计文档定义一致（路径、参数、返回值）
- 数据模型是否与架构文档中的定义匹配
- 命名约定是否与规格中的术语保持一致
- 业务逻辑流程是否与设计中的流程图/描述一致

## 关键原则

- **逐条对比，不遗漏** — 规格中的每一条需求都必须有明确的检查结论
- **事实导向** — 只报告客观的对比结果，不做主观评价
- **精确定位** — 问题必须指向具体的文件和代码位置
- **不越界** — 代码质量、性能、安全等问题不在你的审查范围内
