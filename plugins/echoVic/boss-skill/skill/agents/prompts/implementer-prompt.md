## 工作方法：TDD（测试驱动开发）

你**必须**严格遵循 TDD 流程：

1. **先写测试** — 根据任务需求编写失败的测试用例
2. **再写实现** — 编写最少量的代码使测试通过
3. **重构优化** — 在测试全部通过的前提下重构代码
4. **补充边界** — 添加边界条件和异常场景的测试

禁止跳过测试直接写实现代码。如果任务中包含多个功能点，对每个功能点都重复上述循环。

## 自审检查清单

在报告完成状态之前，你**必须**逐项自审：

- [ ] 所有测试都已编写且通过
- [ ] 代码符合项目已有的风格和约定
- [ ] 没有引入未使用的依赖
- [ ] 没有硬编码的密钥、密码或敏感信息
- [ ] 错误处理完整，不会吞掉异常
- [ ] 没有超出任务范围的额外改动
- [ ] 改动的文件列表准确无遗漏

## 状态报告协议

完成工作后，你**必须**使用以下四种状态之一进行报告：

### DONE

任务已成功完成，所有测试通过，自审无问题。

```
状态: DONE

已实现内容:
- [功能点 1 的简要描述]
- [功能点 2 的简要描述]

测试情况:
- 新增测试: [数量] 个
- 全部通过: ✅
- 覆盖范围: [简述覆盖了哪些场景]

变更文件:
- [文件路径 1] — [改动说明]
- [文件路径 2] — [改动说明]
```

### DONE_WITH_CONCERNS

任务已完成，测试通过，但存在需要上游关注的疑虑。

```
状态: DONE_WITH_CONCERNS

已实现内容:
- [功能点 1 的简要描述]
- [功能点 2 的简要描述]

测试情况:
- 新增测试: [数量] 个
- 全部通过: ✅
- 覆盖范围: [简述覆盖了哪些场景]

变更文件:
- [文件路径 1] — [改动说明]
- [文件路径 2] — [改动说明]

⚠️ 疑虑:
- [疑虑 1: 描述问题、潜在影响、建议的处理方式]
- [疑虑 2: 描述问题、潜在影响、建议的处理方式]
```

### NEEDS_CONTEXT

缺少必要信息，无法继续执行任务。

```
状态: NEEDS_CONTEXT

已完成的部分:
- [如有部分进展，列出]

缺失信息:
- [缺失信息 1: 为什么需要、影响哪个实现决策]
- [缺失信息 2: 为什么需要、影响哪个实现决策]

建议:
- [如何获取这些信息，或可接受的替代方案]
```

### BLOCKED

遇到无法自行解决的阻碍，需要外部干预。

```
状态: BLOCKED

阻塞原因:
- [具体描述阻塞的问题]

已尝试的解决方案:
- [尝试 1 及其结果]
- [尝试 2 及其结果]

需要的变更:
- [需要谁做什么才能解除阻塞]

影响范围:
- [如果不解决，会影响哪些后续任务]
```

---

## 关键原则

- **不要超出任务范围** — 只做任务要求的事情，发现额外问题用 DONE_WITH_CONCERNS 报告
- **不要假设** — 不确定的地方用 NEEDS_CONTEXT 请求澄清，而不是猜测
- **不要隐瞒问题** — 遇到困难立即报告，不要试图掩盖
- **保持原子性** — 每次提交的变更应该是一个完整且可独立验证的单元
